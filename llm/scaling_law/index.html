Some Notes on Pretraining Scaling

June 24, 2026

(This is an on-going blog)

The original “scaling laws” result from Kaplan et al., coauthored by Dario Amodei, showed that language model loss follows predictable power-law trends with model size, dataset size, and training compute across many orders of magnitude.1 Scaling pretraining in practice is not only a question of adding more GPUs. The hard part is deciding what should scale, how to predict whether the training recipe will keep working, and how to keep the system efficient when the job becomes a long-running distributed system.

This note looks at pretraining scale from an ML systems perspective. The main question is: if we start from a small stable run, what has to change before the same training recipe can run on many racks of GPUs? And what are the challenges?

Mental Model #

  1. What Are We Scaling?

    • target: tokens/day, loss, cost/token, MFU, goodput
    • model/data/compute tradeoff

  2. Scaling Methodology

    • small-scale ladders
    • IsoFLOP experiments
    • batch size and LR scaling
    • tokens-per-parameter
    • efficiency gain

  3. Data Pipeline and Mixture

    • filtering, dedup, quality buckets
    • data mixture search
    • contamination and governance
    • dataloader throughput

  4. Model Architecture for Scale

    • dense vs MoE
    • MLA/GQA, MTP
    • expert routing and load balance
    • long-context extension

  5. Distributed Training Framework

    • DP/FSDP/ZeRO, TP, PP, SP/CP, EP
    • topology-aware placement
    • collectives, RDMA, all-to-all

  6. Kernel / Compiler / Runtime

    • compute-communication overlap
    • fused kernels
    • activation recompute
    • FP8/BF16 precision recipe

  7. Cluster Reliability and Goodput

    • MFU vs goodput
    • stragglers
    • hardware health
    • observability

  8. Fault Tolerance

    • async checkpointing
    • deterministic recovery
    • rollback policy
    • restart with changed parallelism

  9. Cost Model

    • GPU hours
    • FLOPs/token
    • cost/token
    • wall-clock planning

The rest of the post expands this outline in one concrete direction: how to reason about hardware capacity and staged scaling on GB200 NVL72 systems.

GB200 NVL72 #

For GB200-class systems, it is more useful to think in rack-scale units than single-GPU units. One NVL72 rack contains 72 Blackwell GPUs connected by NVLink. A rough hardware model is:

  • 72 GPUs per NVL72 rack
  • About 180 GB HBM per GPU
  • About 13 TB HBM per rack
  • About 9 PFLOP/s FP8 peak per GPU
  • About 648 PFLOP/s FP8 peak per rack
  • About 1.8 TB/s GPU-to-GPU NVLink bandwidth per GPU

If the job reaches 40% MFU, one rack provides roughly:

$$ 648 \ \text{PFLOP/s} \times 0.40 = 259 \ \text{PFLOP/s useful compute} $$

For dense Transformer pretraining, a useful approximation is:

$$ \text{training FLOPs/token} \approx 6 \times \text{number of parameters} $$

See the derivation in Transformer: Parameter Count, FLOPs, Memory Usage, and Training Time.

For a 70B dense model:

$$ 6 \times 70\text{B} = 420\text{B FLOPs/token} $$

For a DeepSeek-V3-style MoE with 37B active parameters:

$$ 6 \times 37\text{B} = 222\text{B FLOPs/token} $$

Memory has to be reasoned about differently. Dense models are mostly constrained by active parameters, gradients, optimizer state, and activations. MoE models are constrained by both active compute and total parameter storage. A 70B dense model has about 140 GB of BF16 weights before gradients, optimizer states, and activations. A 671B-total MoE has about 1.34 TB of BF16 weights, or about 671 GB of FP8 weights. Even when raw HBM is enough, training still needs sharding, activation checkpointing, and careful placement.

Scaling From 32 GPUs to Many Thousands of GPUs #

A practical scaling path is:

32 GPUs -> 128 GPUs -> 512 GPUs -> 1K-2K GPUs -> many thousands of GPUs.

At each stage, only scale after the previous stage is stable, efficient, and reproducible.

1. Start With a Stable 32-GPU Baseline #

The goal is to prove that the model, data, optimizer, and training loop are correct.

Use 32 GPUs to establish:

  • Loss curve
  • Tokens/sec
  • MFU
  • Memory usage
  • Global batch size
  • Optimizer settings
  • Activation checkpointing
  • Checkpoint save/load
  • Evaluation pipeline

At this stage, avoid clever scaling tricks. The purpose is to create a clean reference run.

Typical setup:

  • Data parallelism, FSDP, or ZeRO
  • Tensor parallelism set to 1 or 2, if needed
  • Pipeline parallelism off
  • Expert parallelism off unless using MoE
  • BF16 training
  • Simple checkpointing

With 32 GB200 GPUs at 40% MFU, the useful compute is roughly:

$$ 32 \times 9 \ \text{PFLOP/s} \times 0.40 = 115 \ \text{PFLOP/s} $$

For a 70B dense model:

$$ \frac{115 \times 10^{15}}{420 \times 10^9} = 274\text{K tokens/sec} $$

$$ 274\text{K} \times 86{,}400 = 24\text{B tokens/day} $$

This number is mainly useful as a sanity check. At 32 GPUs, the goal is not to predict the final training run; the goal is to establish a clean reference for loss, memory, throughput, and restart behavior.

Do not move on until training can restart from checkpoint and reproduce the same loss behavior.

2. Scale to One Rack / 64-128 GPUs #

The goal is to introduce real multi-node communication.

At this stage, network behavior starts to matter. Measure:

  • All-reduce latency
  • All-gather and reduce-scatter overhead
  • Dataloader throughput
  • Checkpoint time
  • Step-time variance
  • Slow ranks

Tune:

  • NCCL settings
  • Rank placement
  • FSDP or ZeRO bucket sizes
  • Gradient accumulation
  • Microbatch size
  • Activation recomputation

At this point, the bottleneck should be clear: compute, memory, communication, or input data.

One NVL72 rack has 72 GPUs. At 40% MFU, the useful compute is roughly 259 PFLOP/s. For a 70B dense model:

$$ \frac{259 \times 10^{15}}{420 \times 10^9} = 617\text{K tokens/sec} $$

$$ 617\text{K} \times 86{,}400 = 53\text{B tokens/day} $$

For a DeepSeek-V3-style MoE with 37B active parameters:

$$ \frac{259 \times 10^{15}}{222 \times 10^9} = 1.17\text{M tokens/sec} $$

$$ 1.17\text{M} \times 86{,}400 = 101\text{B tokens/day} $$

These are healthy-step estimates. They do not include checkpoint stalls, dataloader stalls, failed nodes, validation, stragglers, or all-to-all imbalance. This is the first scale where the gap between healthy throughput and useful wall-clock progress becomes visible.

3. Scale to 256-512 GPUs #

The goal is to make parallelism topology-aware.

This is where just adding data parallelism often stops working. Start combining:

  • Data parallelism, FSDP, or ZeRO for optimizer and gradient sharding
  • Tensor parallelism for large matrix multiplications
  • Pipeline parallelism if the model no longer fits cleanly
  • Sequence or context parallelism for long sequence memory
  • Expert parallelism if using MoE

Map parallel groups to hardware topology:

  • Tensor parallelism inside NVLink or NVSwitch domains
  • Pipeline parallelism across nodes only if needed
  • Data parallelism across the wider network
  • Expert parallelism placed to reduce all-to-all traffic

At this scale, goodput becomes more important than peak MFU. A lower-MFU run that never crashes can be better than a higher-MFU run that loses hours to checkpoint stalls and failures.

With 512 GPUs, the cluster is about 7.1 NVL72 racks. At 40% MFU:

$$ 512 \times 9 \ \text{PFLOP/s} \times 0.40 = 1.84 \ \text{EFLOP/s} $$

For a 70B dense model:

$$ \frac{1.84 \times 10^{18}}{420 \times 10^9} = 4.4\text{M tokens/sec} $$

$$ 4.4\text{M} \times 86{,}400 = 380\text{B tokens/day} $$

For a 15T-token run, the optimistic healthy-step time is:

$$ \frac{15\text{T}}{380\text{B}} = 39.5 \ \text{days} $$

If goodput is 70%, the wall-clock time becomes about 56 days. This is why the 256-512 GPU stage should test not only step time, but also checkpoint overhead, restart behavior, and steady-state dataloader throughput.

4. Scale to 1K-2K GPUs #

The goal is to harden the training system.

At this size, failures become normal. Add serious infrastructure:

  • Async checkpointing
  • Node health checks
  • Automatic restart
  • Bad node quarantine
  • Straggler detection
  • NaN and loss spike detection
  • Network error monitoring
  • Deterministic recovery

Also validate that training quality still matches smaller runs. Compare:

  • Loss per token
  • Evaluation curves
  • Gradient norms
  • Activation statistics
  • Learning-rate schedule behavior
  • Data sampling correctness

A common failure is thinking the system is scaling because tokens/sec increases, while model quality silently changes because data order, global batch size, or optimizer behavior changed.

At 2K GPUs, the cluster is about 28 NVL72 racks. At 40% MFU:

$$ 2048 \times 9 \ \text{PFLOP/s} \times 0.40 = 7.37 \ \text{EFLOP/s} $$

For a 70B dense model:

$$ \frac{7.37 \times 10^{18}}{420 \times 10^9} = 17.5\text{M tokens/sec} $$

$$ 17.5\text{M} \times 86{,}400 = 1.5\text{T tokens/day} $$

At this point, a 15T-token run looks like about 10 days at healthy-step throughput, but it becomes about 20 days at 50% goodput. A single recurring failure mode can now cost many GPU-days, so fault tolerance becomes part of the training algorithm in practice.

5. Scale to Many Thousands of GPUs #

The goal is to optimize useful tokens per wall-clock day.

At this stage, the job is a distributed system as much as an ML experiment. The system needs:

  • Topology-aware scheduling
  • Parallelism search or careful manual tuning
  • Distributed checkpoint format
  • Fast checkpoint restore
  • Recovery strategy
  • Large-scale observability
  • Capacity planning
  • Storage bandwidth planning
  • Evaluation automation

Optimize for useful training progress, not just healthy-step throughput. Peak FLOPs, MFU, and tokens/sec during clean steps are not enough. At huge scale, the real losses come from startup time, checkpointing, failures, stragglers, validation pauses, and bad placement.

For an 8K-GPU cluster, roughly 114 NVL72 racks:

$$ 8192 \times 9 \ \text{PFLOP/s} \times 0.40 = 29.5 \ \text{EFLOP/s} $$

For a 70B dense model:

$$ \frac{29.5 \times 10^{18}}{420 \times 10^9} = 70\text{M tokens/sec} $$

$$ 70\text{M} \times 86{,}400 = 6\text{T tokens/day} $$

For a DeepSeek-V3-style MoE with 37B active parameters:

$$ \frac{29.5 \times 10^{18}}{222 \times 10^9} = 133\text{M tokens/sec} $$

$$ 133\text{M} \times 86{,}400 = 11.5\text{T tokens/day} $$

DeepSeek-V3 was pretrained on 14.8T tokens, so this style of calculation would make the run look like about 1.3 days in the optimistic healthy-step estimate. In reality, all-to-all overhead, expert imbalance, checkpointing, recomputation, evaluation, failures, and recovery can easily move the wall-clock time by multiples. This is why large labs report goodput as a production KPI: the useful question is not how fast the cluster can step when everything is healthy, but how many useful training tokens it produces per wall-clock day.

The Process Loop #

At every scale, repeat the same loop:

  1. Run a short correctness test.
  2. Run a throughput benchmark.
  3. Profile compute, memory, communication, and dataloader.
  4. Tune parallelism and placement.
  5. Run a medium training segment.
  6. Compare loss and evals against the smaller baseline.
  7. Test checkpoint and recovery.
  8. Increase GPU count only after the above is stable.

The core idea is that scaling pretraining is not one jump from 32 GPUs to 10K GPUs. It is a staged process where each stage exposes a new bottleneck: correctness, communication, topology, reliability, then goodput.
